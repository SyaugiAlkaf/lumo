AMANAH — from the Indonesian for "a trust faithfully kept"

The agent can be tricked.
The money cannot.

Amanah is an on-device treasury agent for cross-border SME payments in USDC on Stellar. The AI can only read invoices — a deterministic policy layer decides, and two Soroban smart contracts enforce. Even a fully compromised model cannot move funds anywhere but the registered supplier, or back home.

01 — THE PROBLEM

Cross-border trade still settles at 1973 speed.

An SME exporter in Yogyakarta paying a fabric supplier abroad waits on correspondent banking and paper letters of credit. Stablecoins fix the rails and AI agents fix the labor — but an agent holding your keys creates a brand-new problem.

3–5 days

Correspondent banking

A payment hops through intermediary banks, each adding time, fees, and FX spread, before a supplier sees a cent.

Letters of credit

Trust by paperwork

Banks intermediate because neither side trusts the other. Slow, manual, and priced for corporates — not for a batik exporter.

1 poisoned invoice

The agent trust gap

Give an LLM your keys and it obeys whoever writes the best prompt. "Our payment address has changed" is all it takes.

02 — HOW IT WORKS

Compromise the AI. The money still can't move.

Amanah splits the agent into three layers with a hard trust boundary: the model is advisory only, and every payment is checked twice more on-chain.

LAYER 1 · UNTRUSTED

The LLM only reads

An on-device model extracts supplier, amount, and reference from invoice text. It holds zero tools, zero keys, zero network access — it cannot act, only report.

  • extraction-only provider seam
  • runs locally, invoices never leave the device
LAYER 2 · DETERMINISTIC

Policy holds the pen

Plain code — not a model — decides. Injection scanner, per-transaction caps, a signed supplier registry. Out-of-policy means a typed refusal and no chain call.

  • NFKC + zero-width injection scan
  • caps · registry · single audit chokepoint
LAYER 3 · ENFORCED ON-CHAIN

The chain is the authority

A policy-signer smart account refuses out-of-policy transactions before submission, and the escrow can structurally pay only the bound supplier — or refund the SME. No third path.

  • __check_auth allowlist + per-tx cap
  • escrow: release → supplier · refund → SME
An adversarial test suite drives a fully compromised LLM — one that obeys attacker text inside the invoice — and proves funds still reach only the registered supplier or return to the SME. The exploit is written; the money doesn't move.
03 — THE DEMO

One command. Five moves. Two of them hostile.

The narrated walkthrough runs the whole spine on a local Stellar quickstart — real contracts, test USDC, no real funds.

  1. Invoice arrives — including a poisoned one
    The agent extracts fields from a real supplier invoice. A fake "our payment address has changed" email is fed through the same pipe — the injection scanner refuses it before any transaction exists. ✕ INJECTION_SUSPECTED
  2. Propose
    Policy checks the cap and the signed supplier registry, then emits an exact create_intent plan. A request_hash binds the proposal to precisely this payment — nothing can be swapped later.
  3. Escrow on-chain
    Funds lock in the Soroban escrow, structurally bound to that one supplier. From this moment no key, prompt, or bug can redirect them.
  4. Attest → release
    The registered oracle attests Shipped (first-write-wins); the escrow releases to the bound supplier — the only address it can pay.
  5. Revert — the failure path
    A second order goes unshipped past its deadline (a real wait, no ledger time-travel). Funds return to the SME untouched; the supplier never sees them.

04 — THE PLATFORM

A trust dial, not a one-off demo.

Every guard is a config flag, every dependency sits behind an adapter seam, and every decision is observable. Full walkthrough in docs/integration.md.

TRUST GUARDS

Turn the dial

Injection · policy · signer · attestation · k-of-n oracle quorum · cosign above threshold · proof-of-compute. Presets: fast, balanced, strict — one line to switch.

ADAPTERS

Any chain, any vendor

Settlement sits behind ChainAdapter · AnchorAdapter · AttestationSource seams, selected by config. Soroban live today; EVM (x402) and real anchor off-ramps are drop-in seams, not rewrites.

OBSERVABILITY

Watch every decision

One event bus feeds Prometheus-style metrics, webhook fan-out, and a live dashboard. Every guard trip and state change is on the record — GET /v1/metrics.

INTERFACES

Call it from any code — or any AI agent

Python SDK, REST with OpenAPI, and an MCP server so any agent can propose payments — while every cap, registry, and injection guard still decides. The caller is never trusted.

from amanah import AmanahClient

client = AmanahClient()                          # or: REST at :8788 · MCP over stdio
decision = client.propose(open("invoice.txt").read())
print(decision.decision, decision.codes)         # proposed | refused | held
